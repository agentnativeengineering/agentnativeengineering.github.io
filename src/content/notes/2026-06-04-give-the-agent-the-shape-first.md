**Why this matters to you.** When an agent must act over a large, structured dataset — a content model, a database, a codebase — your instinct is to give it everything so it misses nothing. That backfires. At a Mastra event (talk uploaded 1 Jun 2026), Rostislav Melkumyan of Sanity, who builds the Content Agent that audits, generates, and bulk-edits content, put it plainly: "If your agent is bad at using tools, you probably don't have a tool problem. You have a context problem" ([talk, Mastra](https://www.youtube.com/watch?v=EEXI_0Jo_bQ)). Sanity is a content backend used by Anthropic, Figma, and Linear; some customers hold tens of thousands of documents, a few up to 10 million ([Mastra Customers, Jan 2026](https://mastra.ai/blog/sanity)). At that size, what is in the window decides whether the agent works.

**The first mistake: more tools.** Their instinct was to add tools, then more tools. In practice it "gets messy very very quickly" — the agent got "less consistent, worse at choosing what to do, and way more likely to pick nearly the right tool." When tools sound similar, "the model spends effort on tool selection rather than the actual user problem." The fix was broader tools, no overlap, better instructions ([talk](https://www.youtube.com/watch?v=EEXI_0Jo_bQ)). An independent writeup records the same: "more tools led to less consistency, worse decision-making" ([ZenML LLMOps Database](https://www.zenml.io/llmops-database/context-first-agent-design-for-content-operations-at-scale)).

**The real fix: shape first, detail on demand.** A real schema is huge — many types, nested objects, references, validations. Dump it all in and "the model sees a lot of detail, but it doesn't see much clarity," so Sanity compresses it to an overview: what types exist, the key top-level fields, and what references what — "a blurry map of your actual content model." In their studio a blog post references an `author`, not a `person`; without the map, an agent asked "how many posts has Sam written?" looks Sam up as a `person`, finds no link, and answers wrong. The compressed view shows author has 843 incoming references from blog post, and "fixed a surprising amount of bad behavior" ([talk](https://www.youtube.com/watch?v=EEXI_0Jo_bQ)). The same idea drives "sets": a reusable handle to a query result with metadata — document count, type breakdown, value ranges — so the agent and the user can check it is the right slice before expensive work runs ([ZenML LLMOps Database](https://www.zenml.io/llmops-database/context-first-agent-design-for-content-operations-at-scale)). Three levels: shape, then exact detail only when needed, then execution.

**Then let code enforce.** Models are good at GROQ — Sanity's open query language ([sanity-io/GROQ, OWFa 1.0](https://github.com/sanity-io/GROQ)) — because so much of it is public, but "close isn't enough." So "the model proposes, but our code actually enforces": they parse the model's query, add the required ID and type fields it may forget, then execute the repaired version, returning a clear error for the model to self-correct ([talk](https://www.youtube.com/watch?v=EEXI_0Jo_bQ)). They also keep one strong main agent over many sub-agents, because handing work off loses the built-up context — "all of that research was gone" ([Mastra Customers, Jan 2026](https://mastra.ai/blog/sanity)).

**What to do tomorrow.** Take the largest structured thing your agent reads — a schema, a table, a long query result — and stop putting the whole thing in the window. Build one tool that returns a compressed overview (types, key fields, counts, relationships) and a second that fetches exact detail for one path on request. Let the agent orient on the shape, drill in only where it must, and run the costly action last.
