**Why this matters to you.** A threat model — the list of attacks a design must defend against, written during security design review — usually goes quiet the moment the review ends. The implementing code lands weeks later in a pull request (PR — one proposed change submitted for review) that nobody compares against it, so controls everyone agreed on can silently never ship. On 2026-06-12, [Dropbox published the measured size of that gap](https://dropbox.tech/security/dropbox-mcp-dash-design-code-security): across 150 design reviews from the prior 18 months, only 12% of implementing PRs linked back to their threat model, the median delay between review and implementation was about five weeks (54% took over a month), and 15% of reviews were filed retroactively, after the code already existed.

**What Dropbox actually does.** It built an [agentic system combining MCP, foundation models, and Dash](https://dropbox.tech/security/dropbox-mcp-dash-design-code-security) — Dash is Dropbox's AI search over its connected content, and MCP (Model Context Protocol) is the standard interface for giving a model access to tools and data. At review time the agent retrieves the threat models relevant to a code change through Dash's MCP server, then the model compares the documented security requirements against the diff — flagging missing controls, contradictions with the approved design, and regressions that static analysis misses. Retrieval mattered as much as judgment: Dash's semantic search [recovered links between 80% of design reviews and their implementing PRs](https://dropbox.tech/security/dropbox-mcp-dash-design-code-security), and 69% of those connections were recoverable only semantically — there was no shared ticket ID or doc link to grep for.

**The idea underneath.** Dropbox's [design principles](https://dropbox.tech/security/dropbox-mcp-dash-design-code-security) are what make this safe to run on real reviews: validate each candidate finding against the actual code before surfacing it, trace every finding back to a source requirement so a reviewer can verify it in one click, keep most findings advisory rather than blocking, and account for context that has gone stale since the review. The advisory choice is the load-bearing one — a model judging "does this diff satisfy that requirement" will sometimes be wrong, and a citation a human can dismiss in seconds costs far less than a false positive that blocks a merge. Dropbox argues the [same pattern generalizes](https://dropbox.tech/security/dropbox-mcp-dash-design-code-security) to privacy, platform API-contract, and compliance reviews — anywhere documented requirements sit disconnected from the workflow that implements them.

**What to do tomorrow.** Measure your own gap before building anything: in your design-review tracker, sample last quarter's reviews and count how many implementing PRs reference them. If your number looks like [Dropbox's 12%](https://dropbox.tech/security/dropbox-mcp-dash-design-code-security), start with retrieval, not generation — expose your design docs to code review (an MCP server over your doc store is one way) so the threat model is at least visible in the PR before you ask a model to judge anything.
