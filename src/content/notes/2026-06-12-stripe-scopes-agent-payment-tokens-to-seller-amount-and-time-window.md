**Why this matters to you.** The moment your agent can spend money, it holds a payment credential — and an LLM planner is non-deterministic, so on a bad day it buys from the wrong place, buys the wrong thing, spends the wrong amount, or uses the wrong credential. If that credential is a raw stored card pasted into a tool call, nothing downstream enforces what it may be used for; the prompt is your only guardrail. On 2026-06-06, Stripe principal engineer Steve Kaliski gave a talk on [building safe payment infrastructure for the autonomous economy](https://www.youtube.com/watch?v=KLSuFPj2ld0) — his framing: agents are already economic actors (they spend tokens at LLM providers today), so the spend path needs real enforcement.

**What Stripe actually ships.** Kaliski walks through [three building blocks](https://www.youtube.com/watch?v=KLSuFPj2ld0) built with partners. Shared payment tokens are credentials scoped to a specific seller, currency, amount, and time window, enforced by Stripe — he demos a charge being rejected for exceeding its mandate — while still passing brand and last-four risk signals so the seller's existing fraud systems keep working. A machine payments protocol, built with Tempo, uses HTTP 402 responses so a tool-call or API request can ask for and settle a micropayment, including on-chain settlement. And the Agentic Commerce Protocol, built with OpenAI, is a standard set of APIs and JSON objects for product catalogs, line items, tax, and fulfillment — agents and sellers exchange structured checkout state instead of scraping web UIs. [Stripe Projects](https://www.youtube.com/watch?v=KLSuFPj2ld0) is built on these primitives, handling recurring or budgeted spend through OAuth-like refresh flows or many scoped tokens; Kaliski declined to share usage volume.

**The idea underneath.** This is [least privilege applied to money](https://www.youtube.com/watch?v=KLSuFPj2ld0). Kaliski's split: discovery and exploration benefit from the planner's non-determinism, but payments, credentials, and checkout require strict determinism. So you don't make the agent deterministic — you make the credential carry the policy. A token that can only pay one seller, up to one amount, inside one time window means a confused, buggy, or prompt-injected agent cannot exceed the mandate, because the enforcement lives at the charge, not in the context window. The same move as token exchange for any non-human identity, applied to a card.

**What to do tomorrow.** Open your agent's checkout or spend tool and find where the payment credential lives. If it is a shared card number or an unbounded API key, replace it with a credential scoped to seller, amount, and expiry — and make sure the scope is enforced server-side by the issuer at charge time, not by instructions in the prompt. The scope is your blast radius; size it to the single task in front of the agent rather than to everything the agent might ever do.
