**Why this matters to you.** Prompt caching is the discount every LLM provider offers when the start of your request is byte-identical to the previous one: the provider reuses the computation for that prefix and charges a fraction of the full price. The catch is that the discount dies the moment any earlier byte changes — and almost every feature you add to an agent (a new skill, a timestamp in the system prompt, a reordered tool list, a sub-agent handoff) changes earlier bytes. You won't see it as a bug; you'll see it as a cost and latency line that creeps up with every release. On 2026-06-06, the founder of OpenClacky, an AI coding agent, [published an engineering account](https://www.openclacky.com/engineering/cache-invalidation-surface) of building the product three times and concluding that every agent feature is a cache-invalidation surface.

**What they actually do.** The [first two architectures failed](https://www.openclacky.com/engineering/cache-invalidation-surface): a RAG-everything design (RAG — retrieval-augmented generation, fetching relevant snippets into the prompt instead of keeping full history) was costly, stale, and missed too much, and a role-based multi-agent orchestration posted good SWE-bench scores but ran slower at roughly 6x the cost, because every handoff between agents rebuilt the prompt and missed the cache. The third generation is a single agent engineered around cache hit rate via seven decisions, including: a byte-frozen system prompt with dynamic state injected as a discardable session-context message, double rolling cache markers that survive history growth and tool-call retries, one invoke_skill meta-tool that isolates sub-agents behind a stable interface, a fixed set of 16 tools, and insert-then-compress — compressing history at idle, while the cache is still warm. Their own [non-rigorous sanity check](https://www.openclacky.com/engineering/cache-invalidation-surface) shows about a 90.6% cache hit rate at lower cost than several competitors.

**The idea underneath.** Caching rewards the longest stable prefix. Whatever sits earliest in the prompt — system prompt, tool definitions — must never change; whatever must change should live as late as possible, in messages the harness can discard without touching what came before. That inverts the usual instinct: instead of asking "what context would help the model," you ask "where can this context live without invalidating everything after it." The post's thesis is to spend the engineering budget on the harness — cache geometry, tool stability, compression timing — and save the intelligence budget for the model.

**What to do tomorrow.** In your harness's prompt-assembly code, capture two consecutive requests from the same session and diff them byte-for-byte from the top. Any churn before the conversation history — a timestamp baked into the system prompt, a tool list whose order isn't deterministic, per-turn state spliced in early — is a cache invalidation you pay for on every turn, and each one is fixable by freezing it or moving it to the end of the prompt.
