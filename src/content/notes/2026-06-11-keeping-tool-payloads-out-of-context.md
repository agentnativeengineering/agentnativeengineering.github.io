**Why this matters to you.** When your agent calls tools through MCP (the Model Context Protocol — the standard interface for wiring external tools to a model), every result comes back into the model's context window. A GitHub search here, a Hacker News fetch there, each returning 15–30 KB of JSON, and the context balloons until you hit per-minute token rate limits and the run dies mid-task — a failure that looks random and costs you twice, because you pay for tokens the model only ever passed along. On 2026-06-10, [Port of Context published a controlled experiment](https://aaif.io/blog/how-port-of-context-eliminated-production-agent-failures/) from their production GTM-intelligence agent — built on the open-source goose framework with Arcade.dev and a custom MCP server, it searches GitHub and Hacker News and posts digests to Slack — where the standard tool loop completed only 56% of tasks and the alternative completed all of them.

**What they actually measured.** The alternative is Code Mode: a sandboxed `execute_typescript` path the team contributed to goose, where the agent writes one TypeScript program that batches its MCP calls instead of round-tripping each call through the model. Across [18 runs on Sonnet 4-5, Opus 4-6, and GPT-5-4](https://aaif.io/blog/how-port-of-context-eliminated-production-agent-failures/), Code Mode delivered 100% of tasks versus 56% for the standard loop, at roughly half the cost per run and about a third of the input tokens (31.8k vs 93.6k) — and [every rate-limit failure occurred in standard-loop runs](https://aaif.io/blog/how-port-of-context-eliminated-production-agent-failures/). The runs also exposed distinct per-model failure profiles: Sonnet passed a phrase query unquoted and confidently reported inflated counts, Opus self-corrected with client-side phrase validation, and GPT-5-4 quoted correctly but twice reported a 50-item array cap as the real total.

**The idea underneath.** The gap comes from [where data lives during the run](https://aaif.io/blog/how-port-of-context-eliminated-production-agent-failures/). In the classic loop, each intermediate payload enters context, gets re-sent on every later turn, and counts against per-minute caps — the model carries data it only needed to forward. In Code Mode those payloads stay in the sandbox; the model sees the program it wrote and the final result, nothing in between. An appendix [compares four Code Mode implementations](https://aaif.io/blog/how-port-of-context-eliminated-production-agent-failures/) — pctx, Cloudflare Code Mode, Anthropic Programmatic Tool Calling, and Pydantic Monty — on sandbox language, pre-flight type-checking, and isolation; the team argues pctx's bring-your-own-LLM design is what made the cross-model comparison possible.

**What to do tomorrow.** Pull one trace from your busiest production agent and total the input tokens that are intermediate tool JSON — results the model never quotes or reasons about, only forwards to the next call. If that pass-through data dominates, move that chain to a code-mode path: [the experiment puts the prize at](https://aaif.io/blog/how-port-of-context-eliminated-production-agent-failures/) about 3x fewer input tokens and the end of rate-limit deaths, and the appendix above is a working shortlist of implementations.
