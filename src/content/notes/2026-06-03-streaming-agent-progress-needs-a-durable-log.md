**Why this matters to you.** If your agent streams tokens to a user one by one, you have quietly taken on two different jobs, and most teams only build the first. At [Temporal's Replay 2026 conference](https://www.youtube.com/watch?v=GzhwnG9TO-c), Eric Ziegler, a software architect at The Washington Post (which builds the [Arc XP](https://www.arcxp.com/about-us/) news platform used by the Boston Globe and others), walked through exactly this. Job one: get a token to the screen fast. Job two: when the connection drops mid-stream — the train goes into a tunnel (my illustration; Ziegler's own example is a phone on a flaky 3G network) — let the reconnecting browser pick up the messages it missed. The first is easy. The second is where teams get bitten.

**What they actually do.** The Post streams over *SSE* (Server-Sent Events — a long-lived HTTP response where the server pushes text events to the browser as they happen). Behind it sits *Redis Pub/Sub* (a worker publishes to a named channel, connected subscribers receive it). Ziegler was blunt about the trade: Redis Pub/Sub is "the purest expression of fire and forget... at-most-once semantics." At-most-once means a message goes to whoever is listening at that instant, and if nobody is, it is dropped with no second try and no replay. For a token that drops as a harmless render glitch, fine. For output the user acts on, or for "my cat on a really flaky network, maybe an old 3G connection," it is not.

**The idea underneath.** SSE already has the resume mechanism: the browser sends a `Last-Event-ID` header on reconnect; the spec leaves it to the server to use that ID to resume where it left off ([WHATWG HTML spec](https://html.spec.whatwg.org/multipage/server-sent-events.html#the-last-event-id-header)). But you can only honor it if something kept an ordered, addressable log of what was sent. Pub/Sub keeps none. So you need a second system behind the pipe that does — historically Redis Streams (same Redis, but it keeps an ordered, replayable log instead of dropping messages) or Kafka.

**What to do.** Tomorrow, separate the two jobs in your own stream. Tag every message with something monotonic — an index or timestamp — and avoid a random UUID (Ziegler's exact warning: a UUID "completely eliminates your ability to" ask for everything since a given point — because replay means "give me everything after position N," and a UUID has no order). Honor `Last-Event-ID` on reconnect, replaying from the log rather than the live pipe. For the log, one option is [Temporal Workflow Streams](https://temporal.io/blog/replay-2026-product-announcements), announced into public preview on May 6 2026: an offset-addressed (each message gets a sequential position number you can ask to resume from), [exactly-once-at-the-execution-layer event channel](https://docs.temporal.io/develop/python/workflows/workflow-streams) — each batch of messages lands exactly once, never dropped or duplicated — and durable across restarts, so the resume cursor is the offset. Ziegler said he had used it for just one day and still recommended it. The exactly-once and offset claims here come from Temporal's docs rather than his talk — he did not vouch for them. Keep the publish step trivial — "you don't want to introduce failure modes at your notification layer." Open building blocks: [sse-starlette](https://github.com/sysid/sse-starlette) for the SSE layer in Python, [Redis Streams](https://redis.io/docs/latest/develop/data-types/streams/) or Temporal Workflow Streams for the durable log.
