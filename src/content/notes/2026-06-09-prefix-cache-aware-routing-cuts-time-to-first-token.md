**Why this matters to you.** Agent traffic is repetitive in a way ordinary web traffic is not: every turn resends the same system prompt, tool definitions, and retrieved documents before the part that actually changed. If your model servers sit behind a standard HTTP load balancer, each request lands on a random GPU that reprocesses that whole shared prefix from scratch — and you pay in time to first token (TTFT, how long the user waits before the first output token appears) and in GPU compute burned on tokens you already processed. On 2026-06-09, Google [published benchmark results for GKE Inference Gateway](https://cloud.google.com/blog/products/containers-kubernetes/gke-inference-gateway-prefix-caching-accelerates-ai-inference/) claiming up to 92.8% shorter TTFT from routing each request to the accelerator that already holds the relevant cache — and, more usefully, a production data point: Snap reports 75–80% prefix cache hit rates in production.

**What the numbers say.** The benchmark, [run by Principled Technologies](https://cloud.google.com/blog/products/containers-kubernetes/gke-inference-gateway-prefix-caching-accelerates-ai-inference/), served Llama 3.1 8B Instruct on identical hardware (eight NVIDIA A100 40GB GPUs) and compared GKE Inference Gateway against a third-party managed Kubernetes service using standard HTTP load balancing: 15.7% higher throughput, 92.8% shorter time to first token, and 62.6% lower inter-token latency. Keep the hedge in view — this is Google's blog reporting favorable numbers about Google's gateway. The stronger signal is [Snap's production report](https://cloud.google.com/blog/products/containers-kubernetes/gke-inference-gateway-prefix-caching-accelerates-ai-inference/): it integrated prefix-cache-aware routing into its Envoy-based service mesh using llm-d and sees 75–80% of requests hitting a warm prefix cache.

**The idea underneath.** When a transformer processes a prompt it builds a KV cache — key/value tensors for every token — and prefix caching keeps that cache for prompt prefixes it has seen before, so a repeated system prompt or shared document costs almost nothing to re-read. But the cache lives on one specific accelerator. Round-robin routing treats model replicas as interchangeable stateless backends, so a request whose prefix is warm on GPU 3 gets sent to GPU 7 and recomputes everything. Cache-aware routing instead tracks which replica holds which prefix and routes on real-time model server metrics instead of round-robin load balancing. The [highlighted use cases](https://cloud.google.com/blog/products/containers-kubernetes/gke-inference-gateway-prefix-caching-accelerates-ai-inference/) are exactly agent shapes: RAG over a fixed documentation set or codebase, and multi-turn chat with a fixed system persona.

**What to do tomorrow.** If you serve open-weight models on Kubernetes behind a plain HTTP load balancer, run one measurement in your load-test harness: send the same long-prefix request twice to one pinned replica, then twice through round-robin routing, and compare TTFT. If the gap is large — for agent traffic it usually is — move routing to a cache-aware layer: GKE Inference Gateway if you are on GKE, or llm-d's prefix-cache-aware routing as Snap runs it.
