**Why this matters to you.** An AI agent decides what code to run at runtime, and some of those decisions come from text it read — a web page, a document, a tool's output. That text can be hostile (prompt injection: instructions hidden in data that the model follows as if you typed them). So if you drop an agent into an ordinary Kubernetes pod with cloud credentials mounted, a misled agent can read those credentials and act as you. A pod keeps workloads from starving each other of CPU and memory; it was never built to stop hostile code inside it from reading what it can reach.

**The fresh example.** At Microsoft Build 2026 (session BRK222, [uploaded 3 June 2026, YouTube](https://www.youtube.com/watch?v=_SzOdHMVPnY)), Kubernetes practitioner Lachlan Evenson warned, running agentic systems on Kubernetes, "don't let those agents run wild because I know security is a massive concern." He released that same day an open-source reference stack he calls KARS (Kubernetes Agent Reference Stack), shown only as an explicit two-minute demo run locally against a cluster — a brand-new reference design to copy the pattern from, with no official support.

**The mechanism worth copying.** Look past the product to one design choice documented in the KARS repo. The agent process runs as a *different operating-system user (UID)* than the router that talks to the model and cloud. The router holds the cloud credential; the agent never receives it. Because the credential lives only where the router's user can reach it, a normal OS file-permission check stops the agent's user from reading it even if the agent is fully taken over (my plain reading of the split; the repo documents the UID separation — agent UID 1000 vs router UID 1001) ([github.com/Azure/kars](https://github.com/Azure/kars), MIT-licensed). Workload identity (a cloud identity scoped to *this* sandbox alone) then gives the agent a least-privilege boundary — it can reach only the few resources it needs.

**This is becoming a shared pattern.** The same idea appears in a vendor-neutral Kubernetes project: agent-sandbox, under SIG Apps. Its maintainers state the problem plainly — "when an AI agent generates and executes code autonomously, security is paramount," so the Sandbox resource "natively supports different runtimes, like gVisor or Kata Containers" for "the necessary kernel and network isolation required for multi-tenant, untrusted execution" ([Kubernetes blog, 20 Mar 2026](https://kubernetes.io/blog/2026/03/20/running-agents-on-kubernetes-with-agent-sandbox/)). It is a Sandbox / SandboxTemplate custom resource, Apache-2.0; per-sandbox network egress appears in its repo only as a desired characteristic the maintainers are still exploring ([github.com/kubernetes-sigs/agent-sandbox](https://github.com/kubernetes-sigs/agent-sandbox)). (My framing: the talk named workload identity and sandbox environments generically; the specific credential split is documented in the KARS repo rather than spoken aloud in the talk.)

**What to do tomorrow.** Pick one agent you run on Kubernetes with cloud credentials mounted in its pod, and verify the exposure first: `kubectl exec` into that pod and try to read the mounted credential or token path as the agent's own process. If it reads, that is your lethal gap. The full fix is a direction to head over time: move that credential behind a small router process that holds the identity and exposes only the calls the agent may make, run the agent under a separate user in its own namespace, and apply a standard Kubernetes deny-by-default NetworkPolicy yourself.
