**Why this matters to you.** If you have run a remote MCP server in production, you have probably paid the sticky-session tax. MCP — the Model Context Protocol, the standard way an agent connects to tools, data, and external services — used to hand each client an `Mcp-Session-Id` and expect every later request to return to the same server instance. Behind a load balancer that forces sticky routing, a shared session store, and a gateway that inspects packets to find the session — exactly the infrastructure that blocks clean horizontal scaling. On 2026-05-27, [Angie Jones of the Agentic AI Foundation argued that MCP is "growing up"](https://aaif.io/blog/mcp-is-growing-up/), and the change underneath that claim is the [2026-07-28 release candidate](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/), locked on 2026-05-21.

**What actually changed.** The release candidate [removes protocol-level session management outright](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/): the `initialize`/`initialized` handshake and the `Mcp-Session-Id` header are gone. Protocol version, client info, and negotiated capabilities now ride in a `_meta` object on every request, with `Mcp-Method` and `Mcp-Name` headers a load balancer can route on without reading the body. As the maintainers put it, [any MCP request can land on any server instance](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/) — so a server that needed sticky sessions and a shared store can run behind a plain round-robin load balancer.

**The idea underneath.** Going stateless does not delete state; it makes state explicit. Rather than hidden session metadata, a tool now [mints an explicit handle — a `basket_id`, a `browser_id` — that the model sees and passes back](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/) as an argument on the next call. Jones is careful that this is not free: [explicit state is easier to inspect, but it still needs good security and lifecycle design](https://aaif.io/blog/mcp-is-growing-up/). The same release narrows the protocol elsewhere, [deprecating roots, sampling, and protocol-level logging](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/) in favor of tool parameters, direct provider APIs, and OpenTelemetry, each with a twelve-month removal window.

**What to do.** If you operate a remote MCP server, stop engineering around sticky sessions and start planning the stateless migration. Audit where your server holds per-session state, and move each piece into an explicit tool-minted handle the client carries between calls. You have a real window: the spec [locks as final on 2026-07-28](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/) with current session-based methods working for a year after — so test the stateless path against your messiest server now, not after the cutover.
