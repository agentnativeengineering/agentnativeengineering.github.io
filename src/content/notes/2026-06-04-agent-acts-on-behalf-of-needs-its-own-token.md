**Why this matters to you.** Your agent calls an internal API for a user. The quick way is to forward the user's login token downstream, so the agent reaches whatever the user can reach. The problem: an agent decides for itself what to call, and some of those decisions come from text it read (a web page, a tool's output) that may be hostile. Forward the user's full-scope token and a prompt-injected agent can do anything the user could — and the downstream log shows only the user, with no record that an agent decided it. On a livestream uploaded 30 May 2026, a solo.io engineer described the old habit plainly: before this, "the best thing that you could do is just hope that proper [RBAC] permissions were set up" ([All Things AI Today, solo.io on YouTube](https://www.youtube.com/watch?v=xBgiBsWvpQ8)).

**The fresh example.** In the same talk the engineer said he had just built "OBO agent identity" for two customers, using a CEL expression — a short policy expression — to "specify the agent name and then say what it can and can't do" ([solo.io on YouTube](https://www.youtube.com/watch?v=xBgiBsWvpQ8)). OBO is *on-behalf-of*: the agent acts for the user but carries its own identity too. The mechanism is a published standard. [RFC 8693 OAuth 2.0 Token Exchange](https://datatracker.ietf.org/doc/html/rfc8693) (January 2020) defines a *token exchange*: a service trades one token for another with a narrower audience and scope, adding an `act` (actor) claim that "provides a means within a JWT to express that delegation has occurred and identify the acting party to whom authority has been delegated" ([RFC 8693 §4.1](https://datatracker.ietf.org/doc/html/rfc8693#section-4.1)).

**The mechanism worth copying.** The exchange mints a new token that keeps the user's identity (the `sub` claim) but sets the `act` claim to the agent and narrows audience and scope to the one downstream API. The RFC frames this as delegation: the agent "still has its own identity separate from" the user, and "any actions taken are being taken by" the agent representing the user ([RFC 8693 §1.1](https://datatracker.ietf.org/doc/html/rfc8693#section-1.1)). So the downstream service can authorize against the user's rights and see that an agent acted; for longer chains, nested `act` claims record each prior actor. Field CTO Christian Posta makes the same case in writing: agents create a gap where systems lose track of "who decided to act" and "who is accountable," and OBO closes it by reissuing tokens "with appropriately constrained permissions" while preserving the user's identity ([blog.christianposta.com, 12 Dec 2025](https://blog.christianposta.com/explaining-on-behalf-of-for-ai-agents/)).

**What to do.** Pick one agent that calls an internal API for a user and check what credential it sends. If it forwards the user's raw token, that is the gap. The full fix is to move the exchange to your edge so the agent never holds a broad credential. The open-source option named in the talk is [agentgateway](https://www.solo.io/products/agentgateway), an Apache-2.0 gateway hosted at the Linux Foundation where "agent identity is a first-class citizen"; it performs the exchange and can deny a call with a CEL rule on the agent's identity. The talk and Posta are from solo.io, which builds agentgateway; RFC 8693 is the vendor-neutral standard underneath, and the pattern works with any gateway that implements it.
